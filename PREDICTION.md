# PREDICTION.md — Ready-Flow 침수예측 모델 피드백 (→ 모델 개발자)

> 대상 API: `POST https://ready-flow-ai.vercel.app/api/predict`
> 작성 목적: 프론트엔드(HomeProtector) 연동 중 발견한 **모델 출력의 구조적 문제**와 개선 방향 공유.
> 연동/통신/스키마는 정상입니다. 아래는 **모델 출력값 자체**에 대한 피드백입니다.

---

## 0. 요약 (TL;DR)

라이브 API를 체계적으로 프로빙한 결과, `flood_probability` 출력이 **확률로서 사용 불가능**한
수준의 결함을 보입니다.

1. **값이 비현실적으로 낮음** — 관측된 최댓값이 약 **0.092 (9%)**. 일강수량 600mm를 넣어도 ~8%.
2. **강우에 비단조(non-monotonic)** — 비가 **더 많이** 오면 확률이 **더 낮아지는** 구간 존재.
3. **사강(dead zone)** — 일강수량 0~80mm에서 출력이 **정확히 0** (도시 침수가 실제로 발생하는 구간).
4. **공간 변별력 거의 없음** — 매우 다른 법정동들이 거의 동일한 값.
5. **시계열 미사용 추정** — 선행 강우(시퀀스 길이)가 출력에 전혀 영향 없음.
6. **출력 양자화** — 사실상 6~7개의 이산값만 등장(모델 표현력 빈약 시사).

가장 치명적인 검증: **2022-08-08 서울 기록적 폭우(신림동 반지하 침수 참사, 일 ≈381mm)** 입력 시
출력이 **0.0765 (8%)**. 실제 사망자가 발생한 최악의 침수일을 "위험 8%"로 평가합니다.

## 1. 재현 방법

```bash
curl -X POST https://ready-flow-ai.vercel.app/api/predict \
  -H 'Content-Type: application/json' \
  -d '{"adm_cd":1162010200,"forecast_daily_rain":[0,0,0,MM]}'
# adm_cd 1162010200 = 관악구 신림동, MM = 오늘 일강수량(mm)
```

## 2. 증거 (라이브 측정)

### A. 강우 민감도 — 오늘 강우만 변화 (신림동, 과거 3일=0)

| 오늘 강수량 | flood_probability |
|---:|---|
| 0 / 5 / 10 / 30 / 50 / 80 mm | **0.0** (전부) |
| 120 mm | 0.0044 |
| 200 mm | 0.025 |
| 300 mm | 0.0765 |
| 400 mm | 0.0765 |
| **600 mm** | **0.0571** ← 400mm보다 *낮음* |

→ 80mm까지 완전 무반응(0), 이후에도 최대 ~8%, 그리고 600mm에서 하락(**비단조**).

### B. 단조성 — 시퀀스 전체 스케일업 (신림동)

| 입력 (mm) | flood_probability |
|---|---|
| `[5,10,20,30]` | 0.0 |
| `[10,20,40,60]` | 0.0044 |
| `[50,80,120,180]` | **0.0765** |
| `[100,150,200,300]` | **0.0187** ← 강우 2배인데 1/4로 *하락* |
| `[200,300,400,500]` | 0.0187 |

→ 강우량을 키우면 위험이 **떨어지는** 명백한 비단조 결함. 물리적으로 불가능한 응답.

### C. 공간 변별력 — 동일 강우 `[30,80,180,381]`, 동만 변경

| 법정동 | flood_probability |
|---|---|
| 관악 신림 / 강남 대치 / 양천 신월 / 중랑 면목 | 0.0765 (동일) |
| 노원 상계 | 0.092 |

→ 저지대/고지대·배수·지형이 크게 다른 동들이 거의 동일. "법정동 단위 예측" 주장과 배치.

### D. 시계열 영향 — 동일 today=381, 길이만 변경

| 시퀀스 | flood_probability |
|---|---|
| `[381]` ~ `[10,30,80,180,381]` (len 1→5) | **모두 0.0765** |

→ 선행 강우(토양 포화·선행강수지수)가 출력에 **전혀** 반영되지 않음.

### E. building_type — `[30,80,180,381]`

| residential / underground / road / null | 모두 0.0765 |

→ 명세대로 미사용(예약 필드). 이건 정상이며 문제 아님(참고용).

## 3. 원인 가설 (개발자 점검 포인트)

1. **심각한 클래스 불균형** — 침수일이 희소하여 모델이 base-rate(≈수 %)로 수렴. 8% 천장이 그 징후.
2. **분포 밖(OOD) 외삽 붕괴** — 학습 데이터에 200mm+ 극한일이 적어, 극한 강우에서 출력이 무너짐
   (300mm↑에서 포화/하락). → B/A의 비단조 설명.
3. **확률 미보정(uncalibrated)** — 출력이 관측 침수 빈도와 매칭되지 않음(이산값 몇 개로 양자화).
4. **피처 비활성/스케일 버그** — 강수 0~80 구간 전 무반응 + 공간 무변별은 입력 피처가
   사실상 죽어 있거나 정규화/클리핑에서 정보가 소실됨을 시사.
5. **시계열 미사용** — 시퀀스 길이 무영향(D). 선행강수 피처가 파이프라인에서 누락된 듯.
6. **모델 표현력 빈약** — 출력 이산값이 6~7개뿐. 매우 얕은 트리/소형 모델 가능성.

---

## 4. 개선 방향 (우선순위)

1. **단조성 제약 도입 (최우선·정확성)** — 강수량에 대해 **단조 비감소** 보장
   (LightGBM/XGBoost monotone constraints 등). "비가 더 오면 위험이 내려가는" 일은 없어야 함.
   회귀 테스트로 단조성 단위테스트 추가.
2. **확률 보정** — isotonic/Platt 스케일링을 관측 침수 빈도에 맞춰 적용. 출력이 실제 확률이 되도록.
3. **클래스 불균형 처리** — class weight / focal loss / 오버샘플링, 적정 base-rate 반영.
   목표: 2022-08-08 신림동 같은 실제 침수일이 **높은** 확률로 나오게.
4. **극한·OOD 대응** — 200mm+ 극한 강우 샘플 보강 또는 물리 사전확률 결합. 600mm가 100mm보다
   낮게 나오는 일 제거.
5. **피처 감사** — 고도/경사/배수용량/불투수면적/**선행강수지수(API)** 가 실제로 연결·가중되는지 확인.
   현재 공간/시계열 무변별은 이 피처들이 inert함을 의미.
6. **검증 리포트 동봉** — held-out 침수 이벤트(예: 2022-08-08)에 대한 PR-AUC/ROC-AUC,
   신뢰도 다이어그램(reliability), 단조성 테스트. **새너티 체크**: 실제 침수일이 상위 확률이어야 함.

## 5. 프론트엔드 현재 대응

연동 측(HomeProtector)은 이 값을 **임의 보정 없이 그대로** 표시하며, 데이터가 없을 때 숫자를
지어내지 않고 `—`로 둡니다. 모델이 개선되면 앱은 **수정 없이 자동 반영**됩니다.
